### 2. Form Components (`components/form/`)

Custom wrappers around UI primitives that add error handling.

#### FormInput

```tsx
import { FormInput } from '@/components/form/form-input'

<FormInput
  id="email"
  {...register('email')}
  placeholder="Seu e-mail"
  error={errors.email?.message}
/>
```

- Extends `React.ComponentProps<'input'>`
- Adds `error?: string` prop
- Wraps `<Input>` with red border on error + `<FormError>` below

#### FormError

```tsx
import { FormError } from '@/components/form/form-error'

<FormError error="Campo obrigatório" />
```

- Renders `<Info>` icon + error message in red
- Returns `null` when no error

#### FormSelect

```tsx
import { FormSelect } from '@/components/form/form-select'

<FormSelect
  error={errors.classId?.message}
  value={selectedClass}
  onValueChange={onChange}
>
  <SelectItem value="id-1">Turma A</SelectItem>
</FormSelect>
```

- Wraps `<Select>` with error display
- Handles placeholder color based on value presence

### 3. Layout Components (`pages/_layouts/`)

#### GlobalLayout

- Renders `<Middleware>` (cookie sync) + `<Outlet>`
- Initializes dayjs with Portuguese locale and relativeTime plugin

#### AppLayout

- Auth guard: redirects to `/sign-in` if not authenticated
- Renders `<Header>` + `<main>` with `<Outlet>`

#### AuthLayout

- Redirect guard: redirects to `/` if already authenticated
- Centered full-screen layout for auth forms

### 4. Application Components

#### Header (`components/header.tsx`)

```
[Logo] / [SchoolSwitcher]                    [PendingInvites] | [ProfileButton]
```

- Logo links to home (`/`)
- Slash separator between logo and school switcher
- Right side: pending invites count + profile dropdown

#### SchoolSwitcher (`components/school-switcher.tsx`)

- DropdownMenu showing all user's schools
- Each school item links to `/school/{slug}`
- "Criar escola" option opens a Sheet (CreateSchool)
- Shows current school avatar + name or "Selecionar escola" placeholder

#### ProfileButton (`components/profile-button.tsx`)

- Shows user name, email, avatar with fallback initials
- Dropdown with "Perfil" link and "Sair" (sign out) button
- Loading state: skeleton for text + avatar

#### Tabs (`pages/app/school/tabs.tsx`)

School sub-navigation (Dashboard, Classes, Alunos, Itens, Prêmios, Membros, Configurações).

- Uses `NavLink` with `isActive` for active styling
- Permission-gated: "Membros" tab only shows if `can('get', 'Member')`
- Loading state: skeleton for each tab

#### BottomNavigation (Viewer only)

- Mobile-style bottom nav bar for the student viewer app
- Fixed position at bottom of screen

## Composition Patterns

### Pattern 1: Page + List + Dialog

The standard CRUD page pattern in the manager app:

```
┌────────────────────────────────────────┐
│  Page (index.tsx)                       │
│  ├── Tabs (school navigation)          │
│  ├── Title + "Create" Button           │
│  ├── List Component                    │
│  │   ├── Search bar (non-functional)   │
│  │   ├── Table with data               │
│  │   │   └── DropdownMenu per row      │
│  │   │       ├── Edit → opens Dialog   │
│  │   │       └── Delete → AlertDialog  │
│  │   └── Empty state text              │
│  └── Create Dialog/Sheet               │
└────────────────────────────────────────┘
```

**Example**: Students page

```tsx
// pages/app/school/students/index.tsx
export function Students() {
  useMetadataSchool('Alunos')
  const { permissions } = usePermissions()
  const [createStudent, setCreateStudent] = useState(false)

  return (
    <div className="w-full space-y-4">
      <Tabs />
      <div className="flex items-center justify-between">
        <h1>Alunos</h1>
        {canCreateStudent && <Button onClick={toggle}>Criar aluno</Button>}
      </div>
      {canGetStudent ? <StudentList /> : <p>Sem permissão</p>}
      <CreateStudent open={createStudent} onClose={toggle} />
    </div>
  )
}
```

### Pattern 2: Dialog/Sheet for Create/Edit

Modals are organized in `pages/app/@dialog/` and `pages/app/@sheet/`:

```
@dialog/
├── awards/
├── items/
└── students/
    ├── create-student/
    └── update-student/

@sheet/
├── classes/
└── schools/
```

**Dialog props convention**:

```typescript
interface CreateStudentProps {
  open: boolean
  onClose: () => void
}

interface UpdateStudentProps {
  open: boolean
  onClose: () => void
  studentId: string
}
```

### Pattern 3: Action + Hook Composition

Form submissions follow this flow:

```tsx
// In page component:
const [{ success, message }, handleAction, isPending] = useAction()

async function onSubmit(data: FormData) {
  await handleAction(
    () => createEntityAction({ body: data }),
    () => {
      queryClient.invalidateQueries({ queryKey: ['...'] })
      onClose()
    }
  )
}
```

### Pattern 4: Permission-Gated UI

UI elements are conditionally rendered based on CASL permissions:

```tsx
const { permissions } = usePermissions()

const canCreate = permissions?.can('create', 'Student')
const canGet = permissions?.can('get', 'Student')

return (
  <>
    {canCreate && <Button>Criar</Button>}
    {canGet ? <List /> : <NoPermission />}
  </>
)
```

### Pattern 5: Loading States with Skeleton

Every data-dependent component implements skeleton loading:

```tsx
// Inline skeleton variant
{isLoading ? (
  <>
    <StudentSkeleton />
    <StudentSkeleton />
    <StudentSkeleton />
  </>
) : (
  data?.students.map(...)
)}

// Skeleton component (defined at bottom of same file)
function StudentSkeleton() {
  return (
    <TableRow>
      <TableCell><Skeleton className="h-8" /></TableCell>
      {/* ... repeat per column */}
    </TableRow>
  )
}
```

### Pattern 6: Empty State

Simple text message when no data exists:

```tsx
{data?.students.length === 0 && (
  <p className="text-muted-foreground p-4 text-center text-sm">
    Nenhum aluno encontrado.
  </p>
)}
```

### Pattern 7: Delete Confirmation

All destructive actions use AlertDialog:

```tsx
<AlertDialog>
  <AlertDialogTrigger asChild>
    <Button variant="ghost">Remover</Button>
  </AlertDialogTrigger>
  <AlertDialogContent>
    <AlertDialogHeader>
      <AlertDialogTitle>Deseja apagar?</AlertDialogTitle>
      <AlertDialogDescription>Esta ação não pode ser desfeita.</AlertDialogDescription>
    </AlertDialogHeader>
    <AlertDialogFooter>
      <AlertDialogCancel>Cancelar</AlertDialogCancel>
      <AlertDialogAction onClick={handleDelete}>Confirmar</AlertDialogAction>
    </AlertDialogFooter>
  </AlertDialogContent>
</AlertDialog>
```

## Styling Conventions

### Color Usage

| Semantic | Token | Usage |
|---|---|---|
| Primary action | `bg-emerald-500 hover:bg-emerald-600` | Create buttons (not using `bg-primary`) |
| Destructive | `bg-destructive` or `variant="destructive"` | Delete actions |
| Muted text | `text-muted-foreground` | Secondary text, placeholders |
| Error | `text-red-500` / `border-red-400` | Form validation errors |
| Background | `bg-background` | Page backgrounds |

> ⚠️ **Note**: Primary action buttons use hardcoded `bg-emerald-500` rather than the `bg-primary` design token. This is the established pattern.

### Spacing

- Page padding: `px-8 py-4` (in AppLayout `<main>`)
- Section spacing: `space-y-4` or `space-y-10`
- Form field spacing: `space-y-1.5` (label → input), `space-y-4` (between fields)
- Button gap: `gap-2` or `gap-3`
- **Button Icons**: Do **NOT** use manual margin classes (`mr-*`, `ml-*`) on icons inside buttons. Spacing is handled by the Button component/layout.

### Typography

- Page titles: `text-xl font-medium`
- Section headings: `text-lg font-medium`
- Body text: default (`text-sm` in most contexts via shadcn/ui)
- Error text: `text-xs font-medium` (in FormError)
- Muted text: `text-sm text-muted-foreground`
- **No Italic Text**: Italic text styling (via `italic` class, `font-style: italic` CSS, or `<em>` tag) is strictly forbidden in this repository.

### Responsive Design

- Manager: Desktop-first. No explicit mobile breakpoints observed.
- Viewer: Mobile-oriented with `BottomNavigation` component.
- Sign-in forms: Responsive widths (`w-full md:w-2/3 xl:w-1/3`).

## Creating New Components

### Checklist

1. Use `export function` (named export, function declaration)
2. Define props via `interface` (not `type`)
3. Use `@/` path alias for imports
4. Use `cn()` for conditional classes
5. Use existing UI primitives from `components/ui/`
6. Place in appropriate directory:
   - Generic reusable → `components/`
   - Form-specific → `components/form/`
   - Page-specific (list, skeleton) → co-locate with page
   - Dialog/Sheet modals → `pages/app/@dialog/` or `pages/app/@sheet/`
7. Follow kebab-case file naming
8. Include loading skeleton if the component fetches data
9. Include empty state for list components
10. **Button Icon Spacing**: Do not use manual margin utility classes (`mr-*`, `ml-*`) on icons inside buttons.
11. **No Italic Text**: Do not use italic styling (`italic`, `font-style: italic`, or `<em>` tag) anywhere in the UI.
